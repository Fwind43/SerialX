# SerialX v0.1.0 Plan

## Version Positioning

`v0.1.0` should be the first clearly usable milestone instead of another incremental patch release.

Compared with the `v0.0.x` cycle, this version should stop treating features as isolated additions and instead close the product loop around:

- stable serial workflows
- reliable workspace behavior
- coherent theme and settings behavior
- basic release readiness

## Goal

Make `v0.1.0` the first version that can be described as a usable desktop serial debugging tool for daily work, not just a fast-moving prototype.

## Release Shape

`v0.1.0` is not planned as a feature explosion release.

The target shape is:

- the current feature set becomes reliable enough for repeated daily use
- the main workflows become easier to operate without hidden edge cases
- release quality becomes more predictable through a clearer validation baseline

## Success Criteria

- Core serial workflows are stable enough for continuous use
- Workspace layout and state restore are predictable
- Theme, appearance, and auxiliary windows behave consistently
- The most common user actions have no obvious interaction regressions
- Release documentation and bug records are complete

## Priority Workstreams

### 1. Core Stability

- Continue fixing serial workflow regressions
- Focus on terminal behavior, selection, search, copy, HEX formatting, disconnected send guards, and send/receive edge cases
- Reduce window lifecycle issues across main window, appearance settings, and converter tool

Planned items:

- finalize terminal copy and select behavior
- finish search interaction details and feedback states
- stabilize HEX display and copy/export consistency
- continue fixing auxiliary window lifecycle regressions

### 2. Workspace Reliability

- Harden split-pane interactions
- Verify pane reorder, resize, tab drag, empty pane cleanup, and selected-port sync
- Ensure workspace snapshot import/export remains compatible with current state shape

Planned items:

- add richer tab operations such as close other tabs and move to a new split pane
- improve workspace snapshot usability with naming and quicker switching
- tighten pane cleanup and selected-port fallback logic
- verify snapshot import/export compatibility with current layout state

### 3. Settings and Theme Completion

- Continue unifying theme behavior across main UI, auxiliary windows, and tool panels
- Fix remaining persistence or restore mismatches in appearance-related state
- Improve settings usability where current flows still feel fragile or inconsistent

Planned items:

- finish theme consistency across main window, appearance window, and converter tool
- polish theme preset management for custom themes
- reduce appearance settings interaction friction
- verify persisted theme and wallpaper state after restart

### 4. Release Readiness Baseline

- Establish a minimum regression checklist for manual verification
- Start identifying the first small batch of automation-friendly checks, even if a full test system is not introduced in this cycle
- Keep README and version documents aligned with shipped behavior
- Maintain a documented local data boundary for persistence, settings import/export, logs, and deferred remote storage

Planned items:

- create a concrete manual regression checklist
- define a narrow set of critical build and smoke checks
- keep README and version docs aligned with real shipped behavior
- maintain the bug log continuously during the cycle

## Must Deliver

These items should define whether `v0.1.0` is ready to ship:

### A. Workspace usability baseline

- split panes remain stable under resize, reorder, tab drag, and port disconnect
- tab operations become sufficient for daily use
- workspace snapshot flow is predictable and recoverable

### B. Terminal usability baseline

- select, copy, search, clear, export, and HEX display are reliable
- text mode and HEX mode no longer behave like separate half-finished paths
- common keyboard and context menu actions feel complete

### C. Appearance and settings baseline

- built-in themes and custom themes behave as one system
- appearance settings window is stable and understandable
- wallpaper and transparency settings do not break window behavior or readability

### D. Release baseline

- manual regression checklist is available and usable
- version docs are current
- major open issues are either fixed or explicitly deferred

## Nice To Have

These can ship in `v0.1.0` only after the must-deliver work is in good shape:

- command workflow polish such as grouping, history, or faster reuse
- better terminal export options
- minor productivity upgrades in workspace management
- small quality-of-life actions in terminal and panel interaction

## Explicitly Deferred

These should stay out of the release unless priorities are deliberately reset:

- protocol parser systems
- plugin or extension architecture
- embedded scripting runtime
- cloud sync or collaboration
- broad platform expansion beyond the current Windows-first target

## First Implementation Batch

The first batch should be used to build momentum without destabilizing the release:

1. workspace interaction polish
2. terminal search / copy / export polish
3. appearance window and theme persistence cleanup
4. manual regression checklist and version documentation cleanup

## Progress So Far

The following items are already landed and now define the current `v0.1.0` baseline:

- terminal right-click menu positioning and visible-area clipping fixes
- terminal `copy current screen` action
- terminal `copy all logs` action
- terminal JSON/TXT export paths
- main-menu export for all-port logs in JSON/TXT
- workspace snapshot save, quick apply, rename, delete, and active snapshot highlight
- workspace quick snapshot timestamps and count visibility
- workspace snapshot restore now refreshes local split-pane state immediately
- split panes now reflow when sidebar width changes
- tab right-click actions for split-pane management
- send history recall per port
- send history keyboard recall with draft preservation
- common command grouping in both the command manager and send dropdown
- command manager search and duplicate actions
- command editor group suggestions for existing groups
- converter window standalone theme variable consistency
- appearance and theme wording cleanup in the settings menu

## Exit Criteria

`v0.1.0` can be considered ready only when:

- the must-deliver items are completed
- the current high-frequency regressions are closed
- builds pass consistently
- the remaining issues are minor enough to defer to `v0.1.1`

## Current Focus

The remaining `v0.1.0` work should stay focused on:

- closing the last high-frequency terminal and send-flow rough edges
- keeping workspace snapshot and pane behavior predictable
- making sure docs and release notes stay aligned with the shipped behavior

## Candidate Features

These are valid `v0.1.0` candidates, but only if they do not compromise the stability goal:

- higher-quality export or log workflow improvements
- command workflow polish
- better terminal convenience actions
- small productivity upgrades in workspace management

## Explicit Non-Goals

The following should not become the center of `v0.1.0` unless priorities change:

- large protocol parsing systems
- plugin architecture
- scripting runtime
- cloud sync
- broad cross-platform expansion

## Development Rules

- Keep development on `codex/v0.1.0-dev`
- Record every confirmed bug fix in `docs/v0.1.0-bug-log.md`
- Prefer shipping a narrower but stable `v0.1.0` over a larger unstable version
- Treat UI changes as product behavior changes and verify them accordingly
