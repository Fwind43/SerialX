# SerialX v0.1.0 Plan

## Version Positioning

`v0.1.0` should be the first clearly usable milestone instead of another incremental patch release.

Compared with the `v0.0.x` cycle, this version should stop treating features as isolated additions and instead close the product loop around:

- stable serial workflows
- reliable workspace behavior
- coherent theme and settings behavior
- basic release readiness

## Goal

Make `v0.1.0` the first version that can be described as a usable desktop serial debugging tool for daily work, not just a fast-moving prototype.

## Success Criteria

- Core serial workflows are stable enough for continuous use
- Workspace layout and state restore are predictable
- Theme, appearance, and auxiliary windows behave consistently
- The most common user actions have no obvious interaction regressions
- Release documentation and bug records are complete

## Priority Workstreams

### 1. Core Stability

- Continue fixing serial workflow regressions
- Focus on terminal behavior, selection, search, copy, HEX formatting, and send/receive edge cases
- Reduce window lifecycle issues across main window, appearance settings, and converter tool

### 2. Workspace Reliability

- Harden split-pane interactions
- Verify pane reorder, resize, tab drag, empty pane cleanup, and selected-port sync
- Ensure workspace snapshot import/export remains compatible with current state shape

### 3. Settings and Theme Completion

- Continue unifying theme behavior across main UI, auxiliary windows, and tool panels
- Fix remaining persistence or restore mismatches in appearance-related state
- Improve settings usability where current flows still feel fragile or inconsistent

### 4. Release Readiness Baseline

- Establish a minimum regression checklist for manual verification
- Start identifying the first small batch of automation-friendly checks, even if a full test system is not introduced in this cycle
- Keep README and version documents aligned with shipped behavior

## Candidate Features

These are valid `v0.1.0` candidates, but only if they do not compromise the stability goal:

- higher-quality export or log workflow improvements
- command workflow polish
- better terminal convenience actions
- small productivity upgrades in workspace management

## Explicit Non-Goals

The following should not become the center of `v0.1.0` unless priorities change:

- large protocol parsing systems
- plugin architecture
- scripting runtime
- cloud sync
- broad cross-platform expansion

## Development Rules

- Keep development on `codex/v0.1.0-dev`
- Record every confirmed bug fix in `docs/v0.1.0-bug-log.md`
- Prefer shipping a narrower but stable `v0.1.0` over a larger unstable version
- Treat UI changes as product behavior changes and verify them accordingly

## Tracking

- Branch: `codex/v0.1.0-dev`
- Bug log: `docs/v0.1.0-bug-log.md`
