## Purpose

This document keeps the next low-risk iteration candidates visible without expanding the `v0.1.0` scope beyond the current README, plan, bug log, and regression checklist.

## Current Decision

Keep the next cycle focused on validation-driven polish rather than broad feature work:

1. Re-check one high-frequency daily-use path from `v0.1.0-regression-checklist.md`.
2. If a defect is found, record it in `v0.1.0-bug-log.md` before changing behavior.
3. Prefer small documentation/checklist alignment when no bounded code fix is obvious.
4. Keep README links and descriptions synchronized with shipped behavior only.

## Validation Boundary

Allowed low-risk checks:

- parse JSON files used by docs or scheduler;
- scan README links and known stale absolute paths;
- list `package.json` scripts;
- inspect `git status` / `git diff` for expected file scope.

Deferred unless the user explicitly approves:

- dependency installation or upgrades;
- release packaging or publishing;
- external API calls;
- secret, `.env`, token, or private local configuration reads;
- hardware-dependent serial-device verification.
